# Track A Treasury Core v0.1 Hardening Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Harden the current Track A Treasury Core with bounded config, minimum inbound values, expiry limits, recipient validation, action-phase evidence, tests, wrappers, and security docs.

**Architecture:** Keep Treasury Core v0.1 narrow: immutable owners, immutable threshold, TON payout proposals only, and plain deposits. Add explicit constants and validation helpers in `contracts/types.tolk`, enforce them in `contracts/Treasury.tolk`, prove behavior with deterministic Acton tests, regenerate generated wrappers through Acton only, and update security docs without claiming mainnet readiness.

**Tech Stack:** Acton 1.0.0, Tolk, Acton emulation tests, generated Tolk and TypeScript wrappers, Markdown docs, PowerShell plus WSL fallback.

## Scope Check

This plan implements `docs/superpowers/specs/2026-05-16-track-a-treasury-core-v0-1-hardening-design.md`.

Do not implement:

- governance or config proposals,
- `configVersion`,
- stale proposal semantics,
- owner addition or removal,
- threshold changes after deploy,
- fee reserve changes after deploy,
- Splitter,
- Jettons,
- Telegram Mini App,
- backend or indexer,
- arbitrary payload proposal execution,
- mainnet release claims.
